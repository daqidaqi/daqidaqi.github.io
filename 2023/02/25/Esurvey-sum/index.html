易卷项目总结

Created2023-02-25|Updated2023-02-25
|Post View:

项目总结

规范思考

  1. 后端接口命名需要一个规范。在开发时，命名接口方法都是想到什么就命名什么，然后就出现这里是 DataByPage，那里是 Data，这里用 Listxx，那里用 getxx，有时是 UserSurveyData，有时 SurveyData，有时写完发现接口名字重复了，就再加个单词，然后 survey 和 form 混用，搞得比较乱，甚至有时写完接口在 Swagger 里总找不到，前端调用接口的时候看名称不能马上知道是指这份问卷的设置数据，还是问卷表单项的数据或者是用户填写完表单的数据以及用户和表单关联的数据。

  2. 开发文档要先动。前期我就有先写文档，把思路理清楚，然后写接口，后面懒了，觉得这个接口应该只有这两个参数，然后后面就发现少了 id，少了分页参数，少了查询条件，这些都是一开始没有先理清楚，导致后面开发在大幅改动接口。

  3. 项目文档流程要分类，要清晰。在写项目功能实现具体方案时，会出现宏观流程与细节流程的混用，导致流程混乱。比如我在写登录功能时，写管理员后台登录，设置用户信息，用户在移动端页面注册后管理员在后端设置角色之类，我有时会把业务流程和代码的具体流程混在一起，写着大体流程不知不觉就写到了代码层面的逻辑。

    个人理解，我觉得可以把流程分为页面布局设计，业务流程（功能实现，不涉及传什么参数到后台，就写填写xx，提交，跳转到 xx 页面，动端可以展示xxx），代码流程（具体代码实现，传递什么参数，怎么校验，查询哪些表…）

  4. 接口要定义自己的规范。现在我的项目接口地址都是想到就写，如 /survey/get/form/data/me 这种都有，就比较不规范，奇怪。我打算定义一个规范，/功能/xxx，把接口写得清晰一点，我记得 apifox 有一个不同公司的开发接口文档，我可以去参考一下。以前我认为 RestApi 优雅，不过他的传参方法有点多，我觉得 get post 就挺好的，也比较简单，甚至全是 post 也好像挺不错的。根据自己的需求来，选择一个适合项目开发的规范就行。鱼皮的视频给了我很大的启发和思考。感谢鱼总。

查看文档的时候有些不方便，浏览器里有很多文档，展开后一屏塞不下，就算折叠后空间小，也存在不方便查找的问题。我想能不能做一个网站，顶部导航栏聚合你需要的文档，点击跳转，下面的容器会跳转到对应文档，顶部导航栏还在。还有一个全局搜索，搜 protable，他会出现 antd，搜 uni.request 会出现 uniapp 的，想法还不成熟，探索中。

开发过程有一些 bug 是比较难解决，特别是刚刚功能测试还可以，然后开发的新功能，然后就报错了，把新功能删掉后还是报错，而且报的错误还是框架里面的属性错误，不能直接定位错误来源，就觉得不可思议，刚刚还可以的。

还是要去多深入理解一些内容，刚刚发现问题了，逻辑没什么问题，不过 v-model 里面不支持三元，出现报错

疑惑

  1. 关系表要不要写 controller 或者 service 或者 mapper，比如用户问卷关系表，查询用户填写的问卷以及问卷数据这条接口是要查询关系表，那我是写在 SurveyService 还是 UserSurveyService，目前我是都写在 SurveyService 里，我感觉比较方便，不过命名有点混乱，UserServey xxx 有时语义不清楚，看不明白

文档规范

  1. 每天 todo 要分开，并记下时间

  2. 流程分为页面布局设计，业务流程，代码流程

  3. 接口规范

接口规范：

名称规范

下面单词作为名称的开头，为行为

GET接口：list 获取数据是列表，get 获取的数据为一条，page 获取数据的分页，数据为列表

POST接口：add 新增，update 修改，delete 删除

规范：行为动作+模块+具体行为（如 UserFillNum 用户填写数量，data 数据）（By 查询某一特殊字段要说明时加，byId一般就不加）+ ByApp 如果接口效果类似，但分为管理系统和移动端网页，就用 ByApp 区分，不用 ByAdmin

具体行为要分清楚，分好类型或子模块或获取的数据

比如我的问卷模块，有问卷设置（问卷的标题，最大填写人数，不包括表单数据），问卷表单结构（问卷表里的 schemaJson），改简单一点，不要说表单了，全部关于问卷，问卷设置，问卷 schema 结构，问卷数据（用户填写的内容）

增删改查有5个接口 add ，delete ，update，get，list，page

开发难点

  1. 对框架不熟悉导致解决问题效率很低。之前前端用的是 Vue，React 是从鱼皮的用户中心，api 项目学的，所以不太熟悉。我在写将 json 传到前端，前端自动渲染问卷表单结构，在前端接收到 json 并赋值给 columns 后，结构没有改变，思考了挺久的，后来发现要将 column 变为响应式变量后结构才会变化。还有写移动端时，在不知道对象属性的名字的情况下怎么调用，后来发现 xxxx[xxx].xxx，可以调用变量。还有 v-model 在使用时不知道不能三元表达式也卡了很久。
  2. 数据结构还需要加强。在 json 传到数据库中，查询拿到 json 后需要转为列表，列表里面是集合，在处理集合是不知道 HashMap 没有按顺序存入，后来发现可以用 LinkHashMap 解决。java 基础比较差，还得二次学习。
  3. 项目经验不足。这个问题也不说算问题吧，现在多做项目就是为了解决经验的问题。在我存用户填写问卷后的数据时，我当时为了节省查询数据的次数以及查询解析的复杂度，我将原本的 key：value 结构改为 key：{value：xxx，value： xx}，将标题和用户的数据存到一个 key 中，等到在做前端的时候，发现它 antd pro request 是 return 一个 {key： value} 的值，这样用 key 来对应渲染，我就知道之前的操作多余了，不过我也没有改过来，后端的解析方法也写好了，目前调用还挺方便，下次在写我会直接存{key：value}。
  4. 未知性。做这个问卷项目时，我是很茫然的。没有接触过，一开始也不知道怎么自动渲染表单结构，也不知道 json 怎么解析为数据结构，还有一个问题，就是表单生成器我一开始只知道有 vue 的开源，所以找 React 的 form-generator 时，我也是感觉很悬，还在博客中找到了，有些东西在没有之前根本想不到有，或者知道有也不知道怎么使用后，之前看视频学习看文档比较少，也看不太明白，现在看文档的水平提高了，终于可以面向文档编程了。

项目收获

  1. 阅读文档水平提高了。现在可以相对以前比较轻松地阅读文档。在 API prop 找自己要的属性，看不懂就到项目里打印一下，有参数，有方法，打印调用一下基本也知道怎么用了，event 也一样调用，method 就用 ref 在方法去调用。做项目，在使用 form-generator 时需要根据文档去配置，能看懂能用才改得了，也是改了挺久才符合自己的需求，也提高了看文档的效率。

  2. 写文档水平有所提升。我的语言表达能力有所欠缺，写文档的时候没有逻辑，思路跳来跳去，想到什么写什么，语句还很多，这是语文课欠下的债啊。是时候来你补了，现在我也希望通过多写博客，相关技术文档来提高自己的表达能力。做项目的流程也比以前清晰了，怎敢想到之前做项目没有文档，想到写什么就写什么，项目没做成很有可能是因为没有写文档。

  3. 做项目比以前有经验了，也有自己的规范。之前做项目没有方法名称规范，接口规范导致代码看得很乱，效率也有所下降。后面我自己定义了一套简易的规范，后续遇到问题再把规范完善一下，有规范在开发是效率是有所提升的，我以前在想方法名字都是想到什么写什么，没有结构层次，现在我按照 行为动作+模块+具体行为+By多端 ，这样我再写名字的时候就只需要考虑具体行为的表达单词，还有，在swagger 里面找的时候也更加方便，明确。

  4. 不会太纠结 ui 了，现在主打简洁、层次。ui 问题是绕不过去的坎，尽管自己已经在暗示不要在管 ui 了，可以有时就是想调一调。对于 ui 的看法，我在鱼皮那些启发到的，不用太注意，我不是 ui 设计师，能把 ui 写好就怪了，先把文字什么的先堆上去，在拿几个大厂的网页去模仿就可以了。我现在写前端 ui 的思路是，用卡片将内容文字包起来，展示出来，等功能没问题了，我再拿别人好看的 ui 去模仿。
    这里有个小技巧，对于我这种不太会写样式的人帮助挺大了，我写 ui 一般追求简洁（写起来也简单点），我的颜色一般使用后黑白灰，加卡片，阴影，圆角，然后就是重要的层次，我再观察别人的 app 时发现他们颜色也不是很多，但又简洁又好看，我觉得是他们层次分的好。我们可以学习，在不同内容上用相近的颜色，突出重点，次要的浅一点，也比较容易开发又不会太丑。

项目解决

  1. 用户中心改造，antd pro 使用 v5。后端，在 springboot-init 中补充了一些内容，完善自动登录以及一些小细节。在用 antd pro 时将 api 平台 umi4 改为 umi3，一方面考虑到对 React不熟悉，umi3 生态好，容易找问题的解决方法，另一方面当然是鱼总推荐啦。在改造过程，还是有挺大的变化，不过在一遍看文档一遍对比后改造完成。
  2. 后端传递 schemaJson ，前端 and pro 动态渲染标题和内容。一开始没有用响应式变量 columns ，也琢磨了挺久的。
  3. 对表单生成器的改造。一遍看文档一遍配置属性。

个人感受

之前看 b站的项目确实学到了不少东西，不过一些开发项目的思路过程，看文档的方法也有所欠缺，很多知识都是准备好了，也有 ppt，思路也在 ppt 上，好处就是知识都是干货，体系大。在鱼总这，我学到的更多的是一些开发思路，如何去写文档，如何看文档，找 bug，测试，一些神奇插件的使用，项目的部署等等，收获很多，也给了我很多启发，包括之后对流程的拓展，找 bug 的思路都有很大帮助。同时，星球的小伙伴也给我很大“卷力”，都是大佬，都是大佬。

到目前加入星球已经一个月，这一个月的时间里我先后完成了用户中心，伙伴匹配系统，api 开放平台，国庆训练营前半部分。问卷系统的项目文档结构思路也都参照了国庆训练营文档。问卷系统开发有半个月了，从项目信息，项目调研，技术选型，需求设计，规划时间到开发流程，具体实现到最后部署上线，都详细地思考并记录在文档里面，帮助我理清思路流程，切实提高了开发效率。
